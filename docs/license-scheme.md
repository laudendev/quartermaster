# License Scheme

Normative specification for the signed license format, the
activation state machine, and the threat model this design accepts,
rejects, or explicitly does not attempt to solve.

## Payload layout (v1, permanent ABI)

33 bytes, big-endian, fixed-width. Followed by a 64-byte Ed25519
signature over exactly these 33 bytes. 97 bytes total.

| Offset | Field | Type | Notes |
|---|---|---|---|
| `[0]` | version | `u8` | Always `1` for this spec. Stamped by `Issue`; caller input is ignored. `Verify` rejects any other value before checking the signature. |
| `[1:17)` | license_id | `[16]byte` | Random UUIDv4, generated by `license.NewID()`. |
| `[17:21)` | product | `[4]byte` | ASCII, exactly 4 characters. No shorter, no longer, no padding. |
| `[21:23)` | major_ver | `u16` | Product major-version ceiling. Not the payload format version — see below. |
| `[23:25)` | seats | `u16` | See "Seats" below. Never stored as `0`. |
| `[25:33)` | issued_at | `i64` | Unix seconds, UTC. |

License file = `base32(payload || signature)`, standard alphabet,
no padding, dash-inserted every 5 characters for display.

### Version vs. major_ver — these are not the same field

`version` describes the **byte layout of the payload itself** — it
changes only if a future spec moves offsets, changes field widths,
or adds/removes fields. It exists so `Verify` can refuse to
misinterpret a payload it doesn't know how to parse, rather than
silently reading garbage at the wrong offsets.

`major_ver` describes **the product**, not the wire format — it's
informational data the product's own code can use however it likes
(e.g. "this license was issued for v1.x of the app, refuse to
activate against a v2 binary"). Bumping `major_ver` is a normal,
frequent product decision. Bumping `version` is a rare, disruptive
event that requires new parsing code in `Verify` while preserving
every previous version's parsing path forever, since old licenses
must keep verifying.

### Why the version check happens before the signature check

An unrecognized version byte means the rest of the offsets in this
document may not describe the payload correctly at all — a future
version could move fields, change widths, or add a field. Attempting
`ed25519.Verify` against bytes parsed under the wrong assumed layout
is meaningless. `Verify` therefore reads and checks the version byte
first, and only calls `ed25519.Verify` if the version is recognized.

## Seats

- `0` supplied at issuance is coerced to `1`. A license always
  represents at least one seat.
- The webhook layer (`stripe.go`) enforces the real-world ceiling:
  metadata `seats` values above 24 are rejected outright — the
  checkout is acknowledged (so Stripe doesn't retry) but never
  enqueued, and the rejection is logged clearly, since a value that
  far out of range is treated as a signal that something upstream
  (a Checkout misconfiguration, a tampered request) is wrong, not as
  a large but legitimate order.
- `license.Issue` itself does not enforce an upper bound — the cap
  is a business rule enforced once, at the point of intake, not a
  property of the wire format. The format can technically represent
  up to 65,535 seats (`uint16`); nothing between the webhook and the
  signer re-checks the 24 ceiling. If that boundary is ever moved to
  a different call path (a future admin tool that issues licenses
  directly, for example), the 24-seat check needs to move with it or
  be duplicated there.

## Product code

Exactly 4 ASCII characters, enforced in `license.payload()`.
Characters may repeat (e.g. `BOOK` is valid despite repeating `O`).
There is no uniqueness constraint across the 4 characters — only a
length constraint.

## Fingerprint

`SHA-256(machine_id || product_code)`, hex-encoded.

- Linux: `/etc/machine-id`
- macOS: `IOPlatformUUID`
- Windows: `MachineGuid`

An OS reinstall changes the fingerprint by design — this is treated
as a new machine, not a bug. The support path for a legitimate
reinstall is a human-gated deletion of the stale activation row
after purchase verification; there is no self-service reset in v1.

## Activation state machine

A license, once issued, moves through activation state per
`(license_id, fingerprint)` pair — not per license alone. A single
license can have zero, one, or many active fingerprints, up to its
seat count.

- **Unclaimed → Active**: `POST /license/activate`. Succeeds if the
  license isn't revoked and either this fingerprint is already
  active (idempotent — see below) or a seat is free.
- **Active, same fingerprint, repeated activation**: succeeds,
  no-op. This is the normal path for every app launch after the
  first — the client always calls activate, and the server
  recognizes it's the same machine rather than treating it as a new
  seat claim.
- **Active → Unclaimed (this fingerprint only)**: `POST
  /license/deactivate`. Frees the seat. Does not affect other
  fingerprints on the same license.
- **Seat exhausted**: a *new* fingerprint attempting to activate when
  `CountActivations >= Seats` and this fingerprint isn't already
  active — refused with `409 Conflict`.
- **Revoked**: any activation attempt against a license with a
  `revoked` row is refused with `403 Forbidden`, regardless of seat
  availability. Revocation is set manually today (chargeback
  response), not automatically.

## Threat model

| Concern | Status | Notes |
|---|---|---|
| Forging a license without the private key | Prevented | Ed25519 signature over the full payload; infeasible to produce a valid signature without `signing.key`, which never leaves the signer machine. |
| Tampering with a genuine license (e.g. editing seat count) | Prevented | Any change to the payload invalidates the signature over it. |
| Sharing one key across many unrelated machines (Discord scenario) | Prevented, bounded by seat count | First N activations (N = seats) succeed; the N+1th fingerprint is refused. Existing activated machines are unaffected. |
| Reading/decoding a license's contents | Not prevented, not a goal | Base32 and the payload layout are fully public. Nothing in this scheme relies on the format being secret — only the private key is secret. |
| Forging a Stripe webhook request | Prevented | HMAC-SHA256 over the raw body, keyed with the webhook signing secret, with a 5-minute replay window. Requires possession of the secret, which is issued by Stripe and never transmitted in any request. |
| Independent confirmation that a webhook's transaction ID exists at Stripe | Not implemented, deliberate | Authenticity relies solely on HMAC verification of the signed payload, the standard trust boundary for Stripe webhooks. No callback to Stripe's API re-confirms the session server-side. |
| Webhook signing secret leaking | Not prevented by the license scheme itself | A leaked secret allows unlimited valid-looking sign requests until rotated. No request-volume rate limiting is implemented today; see `docs/operations.md` for the monitoring and rotation plan. |
| A crafted seat count far outside realistic range (e.g. `500`) arriving via webhook metadata | Prevented | Rejected outright above 24, logged as a likely integrity problem, never enqueued. See "Seats" above. |
| A future payload version being misparsed under the current version's offsets | Prevented | `Verify` checks the version byte and refuses unrecognized versions before attempting to read any other field. |
| Base32 encoding/decoding "hiding" license contents | Not a goal, not relied upon | Base32 is a reversible, public representation with no secrecy property. Trailing bits in the last character of a non-block-aligned payload length carry no semantic weight and cannot be used to construct a different valid payload — see `license_test.go`'s tamper test, which flips every raw byte (not every encoded character) specifically to avoid a false sense of coverage from base32's own padding slack. |
| Refund or chargeback after a license is issued and activated | Not prevented, accepted cost | The signature doesn't know about payment status. A `revoked` flag exists for chargebacks specifically, checked on every activation attempt, but a machine already activated before revocation keeps working until it needs to reactivate. |
| Honest resale via deactivate → hand off → reactivate | Not prevented, treated as legitimate | Equivalent to reselling physical media; the platform tracks concurrent machine count, not ownership history. |
